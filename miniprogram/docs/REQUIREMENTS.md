## 1. 产品概述

### 1.1 定位

**遗产通**是一款面向中国用户的**遗产与继承法领域知识型小程序**，以《中华人民共和国民法典》继承编为核心，聚合法条原文、通俗解读、真实判例、实用工具与 AI 辅助问答，服务于**法律小白**与**专业人士**的快速检索与学习场景。

### 1.2 价值主张

- **可信**：法条与结构对齐官方文本，案例来源于公开裁判文书维度整理。
- **易懂**：每条法条配备白话说明、生活化示例与关键词，降低理解门槛。
- **可行动**：案例多维筛选、法条互链、工具箱与 AI 助手缩短「从问题到依据」的路径。
- **可沉淀**：收藏、笔记、浏览历史均在本地管理，便于反复查阅。

### 1.3 范围说明（V1.0.0）

| 在范围内 | 不在范围内（可后续版本） |
|---------|------------------------|
| 离线 JSON 数据集展示与检索 | 用户账号体系、云端同步 |
| 本地收藏/笔记/历史 | 付费法律咨询闭环 |
| DeepSeek 对话式 AI（需网络） | 替代执业律师出具法律意见 |
| 全局搜索与法条互链 | 全量裁判文书实时抓取 |

## 2. 目标用户画像

### 2.1 类型一：法律小白（普通用户）

| 维度 | 描述 |
|------|------|
| 特征 | 无系统法学训练，遇继承、遗嘱、分家等场景需要「看得懂的说法」 |
| 动机 | 快速了解权利义务、常见流程、风险提示 |
| 期望 | 专题导读、FAQ、案例故事化摘要、术语词典、AI 用通俗语言回答 |
| 成功标准 | 10 分钟内找到与自身问题相关的法条或案例入口，并理解核心结论 |

### 2.2 类型二：法律从业者（律师、法务、法学生）

| 维度 | 描述 |
|------|------|
| 特征 | 熟悉法律检索，重视条文编号、裁判要旨、标签与依据链 |
| 动机 | 移动端碎片时间核对条文、浏览类案、向客户展示依据 |
| 期望 | 精准法条页（含原文与延展字段）、案例法律依据跳转、高效搜索 |
| 成功标准 | 通过搜索或章节导航在 3 次点击内打开目标法条/案例详情 |

### 2.3 类型三：存在或潜在遗产纠纷的用户

| 维度 | 描述 |
|------|------|
| 特征 | 情绪与利益敏感，需要客观信息与边界清晰的自助工具 |
| 动机 | 了解「可能」的法律后果与常见争议点，为协商或聘请律师做准备 |
| 期望 | 类案过滤（类型/地区/年份）、法条互链、AI 仅作信息辅助且提示非法律意见 |
| 成功标准 | 获得结构化信息（法条+类案），并明确产品**不构成法律意见**的边界 |

### 3.2 首页（`pages/index/index`）

| 功能点 | 描述 | 验收标准（Acceptance Criteria） |
|--------|------|--------------------------------|
| 搜索入口 | 点击进入全局搜索页 | 任意机型点击后 1s 内进入搜索页；返回栈正常 |
| AI 入口 | 进入 AI 法律助手 | 跳转子包 AI 页成功；弱网下有加载提示或可重试 |
| 今日案例 | 基于日期算法从案例库选取 1 条展示 | 案例总数>0 时每日稳定展示一条；点击进入对应详情 |
| 热门/精选问题 | 展示 FAQ 列表（如前 5 条） | 与 `faq.json` 数据一致；点击可进入关联专题或预期落地页 |
| 热门问题快捷标签 | 快捷 chip 跳转知识专题 | 每个 chip 绑定有效 `topicId`，跳转专题详情无 404 |
| 知识专题网格 | 展示专题卡片（如前 6 个） | 与 `topics.json` 一致；点击进入 `topic-detail` |
| 适配 | 状态栏、胶囊按钮区域 | 顶部安全区与微信原生导航不重叠 |

### 3.3 法律库（`pages/laws/laws` + `subpkg/law-detail`）

| 功能点 | 描述 | 验收标准 |
|--------|------|----------|
| 法条规模 | 民法典继承编相关条文 | **45 条**法条数据完整可打开 |
| 章节分类 | 按 `chapter` 分组列表 | 分组与数据字段一致，折叠/展开或滚动浏览可用 |
| 关键词高亮 | 列表或详情中突出关键词 | 与 `keywords` 字段展示一致（具体 UI 以实现为准） |
| 官方原文 | `originalText` 展示 | 详情页完整展示原文，换行与排版可读 |
| 延展内容 | 白话解释、示例、释义、沿革等 | 各字段有数据时均展示；缺失时 gracefully 隐藏 |
| 法条详情 | 子包页按 `id` 打开 | `id` 非法时友好提示，不白屏 |

### 3.4 案例库（`pages/cases/cases` + `subpkg/case-detail`）

### 3.6 AI 法律助手（`subpkg/ai/ai`）

| 功能点 | 描述 | 验收标准 |
|--------|------|----------|
| 模型 | DeepSeek Chat API | 请求成功时流式/非流式以实现为准，失败有错误提示 |
| 上下文对话 | 多轮历史参与请求 | 重进小程序可恢复近期会话（本地存储上限以设计为准） |
| 法条关联 | 回复中的「第XXXX条」可点击 | 阿拉伯数字四位条号在继承编范围内可跳转法条页 |

| 领域约束 | System Prompt 限定继承编 | 非法条领域问题时简短回答并引导回继承法主题 |
| 安全提示 | 敏感信息 | 产品内提示用户勿输入身份证号等；隐私政策可触达 |

### 3.8 全局搜索（`pages/search/search`）

| 功能点 | 描述 | 验收标准 |
|--------|------|----------|
| 跨域检索 | 法条 / 案例 / FAQ / 专题 / 术语 | 各类型均能出现在结果列表并带类型角标 |
| 触发条件 | 查询长度 | **≥2** 字符触发检索（与 `app.search` 一致） |
| 案例懒加载 | 案例数据 | 未加载 `cases.json` 时案例不参与匹配；进入搜索前或 onLoad 触发加载策略与产品一致 |
| 空结果引导 | 无结果时 | 展示空态，并引导用户使用 **AI 法律助手**（按钮或文案） |
| 防抖 | 输入性能 | 输入过程不因全量扫描卡顿（实现侧 debounce） |

### 3.9 法条互链

| 功能点 | 描述 | 验收标准 |
|--------|------|----------|
| 条号格式 | `第1127条` 等形式 | 正文中匹配四位阿拉伯数字条号，且在 **1119–1163** 范围内生成可点击链 |
| 中文数字 | 「第一千一百二十七条」等 | `CN_NUM_MAP` 覆盖范围内可解析为对应 `id` 并支持提取/跳转（以 `law-link.js` 为准） |

## 4. 非功能需求（NFR）

| 类别 | 指标 | 说明 |
|------|------|------|
| 性能 | 首屏可交互 **< 2s**（中高端机、4G 良好网络） | 以微信性能面板与人工抽样为准；大 JSON 懒加载计入 |
| 包体积 | 主包 **< 2MB** | 遵循微信平台限制；大体积数据放分包 |
| 分包 | 子包合理拆分 | 详情与大案例数据置于 `subpkg`，见技术设计 |
| 兼容性 | 微信 **7.0+** / 基础库按 `project.config` 设定 | 真机矩阵抽样：iOS/Android 各至少 2 款 |
| 可用性 | 弱网与失败 | 请求超时、存储满等异常有提示，不静默失败 |
| 可维护性 | 数据与代码分离 | 内容迭代以 JSON 为主，减少发版成本 |

## 6. 与 Android 版（或原生 App）差异说明

> 若团队另有 Android 客户端，本小程序在**同一内容资产**前提下，侧重微信生态与平台约束下的工程取舍。

| 维度 | 小程序实现要点 |
|------|----------------|
| 加载策略 | **分包加载**：主包仅保留 Tab 与公共逻辑；大体积 `cases.json` 置于子包，按需 `require` |
| 预下载 | **`preloadRule`**：从首页预取子包，降低首次打开详情/AI 的等待 |
| 按需注入 | **`lazyCodeLoading: requiredComponents`**，减少启动期 JS 解析量 |
| 存储 | `wx.setStorageSync` 配额与键名策略；无 SQLite 时的列表裁剪（如历史 **100** 条） |
| 能力边界 | 分享、广告组件、开放能力以微信审核规则为准；AI 调用需配置合法域名 |
| 更新节奏 | 前端资源随小程序发版；**无热修复字节码**，内容更新依赖提审或通过云控配置（若后续引入） |
